Real‑Estate Platform – Minimal MVP → Full DevOps

Based on your refined requirements, I’ve restructured the project into phased chunks, each with a clear goal and verification checkpoint. The plan now respects:

Backend as the single contract source (OpenAPI YAML) that can generate both a read‑only BFF and direct API clients.

BFF is optional & removable – the frontend can call the backend directly via generated TypeScript clients.

Domain package enforced as the sole location of business rules.


AI context served via MCP documentation servers for live accuracy.

Testing after each integration before moving on.

Free DevOps pipeline from day one.

0. Master Architectural Principles (Unchanged & Strengthened)
Clean Architecture – Domain layer (packages/domain) contains ALL business logic; frameworks live in infrastructure.

Contract‑First, Backend‑Owned – A single OpenAPI 3.1 YAML file defines every endpoint, DTO, and validation. This file is the source of truth.

Temporal API – All date handling in domain must use Temporal, never Date.

Optimistic UI – Every mutation uses optimistic updates; all lists paginated.

Testing Pyramid – BDD → Unit (domain) → Integration (API) → E2E (smoke).

AI Governance – Rules enforced by linters + MCP servers, not just prompt context.

Create specs/openapi.yaml that describes the backend REST API for Property Listings.

yaml

info:
  title: Real Estate API
  version: 1.0.0

Include all schemas (DTOs) and validation rules (via JSON Schema).

Validation integration: Zod schemas automatically derived from JSON Schema (using json-schema-to-zod). This guarantees backend validation matches the contract.

No business logic outside domain. Dates serialised as ISO 8601 strings.

STAGE 4: Optional Read‑Only BFF (Goal: GraphQL Middleware Layer That Can Be Removed Without Breaking Anything)
Now we introduce the BFF as an add‑on, not a dependency.

Chunk 4.1 – Generate GraphQL Schema from OpenAPI
Use @graphql-mesh/openapi or openapi-to-graphql to dynamically create a read‑only GraphQL schema from the running backend. This is a thin layer with no mutations; it only maps REST endpoints to queries.

Alternative approach for static generation:
If you want a static schema, use a tool like swagger-to-graphql (build time) to produce a GraphQL SDL file that can be served. The schema remains in sync because it’s derived from the YAML.

Chunk 4.2 – Stand‑Alone BFF Server

A separate apps/bff (GraphQL Yoga) that imports the generated schema (from the contract) and forwards queries to the backend.

No custom resolvers containing logic; pure pass‑through with optional field selection optimisation.

The BFF server is deployed independently and can be turned off at any time.

Chunk 4.3 – Feature‑Flaged Frontend Integration

TanStack Query can be configured to use either the direct REST client or the GraphQL endpoint based on an environment variable.

The REST client (from OpenAPI) remains the default; GraphQL is an experiment.

Verification:

With USE_BFF=graphql, the frontend queries listings via GraphQL and displays same data as REST.

Switch back to USE_BFF=rest and all functionality works – proving the BFF is truly optional.

Performance test: compare response times and payload sizes; this data will decide if the BFF stays permanently or is dropped.

Implementation:

Use @modelcontextprotocol/server with resources:

7. tRPC vs GraphQL as Optional BFF – Comparison
Criteria	tRPC BFF	GraphQL BFF (generated from OpenAPI)
Contract Source	Code‑first: BFF router definitions become the contract. Must manually mirror backend endpoints.	Generated schema from OpenAPI YAML; stays in sync with backend automatically.
Optionality	If BFF removed, frontend loses typed client (unless you also generate from OpenAPI).	Frontend can fall back to REST client from same OpenAPI spec without change.
Type Safety	Excellent (end‑to‑end types shared) but only if both frontend and BFF use the same procedures.	Good via codegen from schema; typing is per‑operation but still robust.
Developer Experience	Very simple API (no AST parsing); just call bff.listings.all().	Slightly more complex (graphql‑request) but tools like graphql‑codegen generate hooks.
Performance	Very fast; no query parsing overhead.	Has parsing cost; can suffer from N+1 if not careful (but read‑only BFF can batch calls).
Flexibility	Limited to the procedures defined. No field selection; always returns full DTO.	Clients can request only needed fields, reducing payload size.
Tooling & Ecosystem	Mature for Next.js; TanStack Query integrates well.	GraphQL ecosystem huge, but adds operational complexity (schemas, stitching).
Maintainability	BFF must be manually updated when backend changes (breakages caught by TypeScript).	Schema generation ensures drift is caught at CI. BFF almost maintenance‑free.
Recommendation:
Given your hard requirement that the BFF be optional and derived from a single YAML contract, GraphQL (auto‑generated) is the clear winner. tRPC would require duplicating the contract logic, breaking the “single source of truth” principle. The GraphQL layer can be generated with zero manual coding and can be turned off without any client‑side code changes (the same REST client works). Start with direct REST calls; add the GraphQL BFF later if you need field selection or query aggregation, knowing it’s purely a read‑side optimisation.
