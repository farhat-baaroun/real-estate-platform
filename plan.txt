0. Master Architectural Principles
Clean Architecture – packages/domain contains ALL business logic; frameworks live in infrastructure.

Contract‑First, Backend‑Owned – A single OpenAPI 3.1 YAML file is the source of truth for every endpoint, DTO, and validation rule.

Temporal API – All date handling in the domain layer must use Temporal, never Date.

Optimistic UI – Every mutation uses optimistic updates; all lists paginated.

Testing Pyramid – BDD scenarios → Domain unit tests → API integration tests → E2E smoke tests.

AI Governance – Rules enforced by linters + live MCP documentation servers, not just static prompt files.

STAGE 2: Backend – Contract‑First with OpenAPI, Auth & Reusable Pagination
2.1 Canonical OpenAPI YAML
Create specs/openapi.yaml with shared pagination, filtering, and limit components that are reused on every list endpoint.

yaml
components:
  parameters:
    Page:   { in: query, name: page,  schema: { type: integer, minimum: 1, default: 1 } }
    Limit:  { in: query, name: limit, schema: { type: integer, minimum: 1, maximum: 100, default: 20 } }
    Sort:   { in: query, name: sort,  schema: { type: string, enum: [createdAt_asc, createdAt_desc, price_asc, price_desc] } }
  schemas:
    PaginationMeta:
      type: object
      properties:
        totalCount:   { type: integer }
        pageCount:    { type: integer }
        currentPage:  { type: integer }
        perPage:      { type: integer }
    ListingFilter:
      type: object
      properties:
        minPrice:  { type: number }
        maxPrice:  { type: number }
        bedrooms:  { type: integer }
        status:    { type: string, enum: [draft, published, archived] }
    ListingPage:
      type: object
      properties:
        data: { type: array, items: { $ref: '#/components/schemas/Listing' } }
        meta: { $ref: '#/components/schemas/PaginationMeta' }

Why shadcn/ui v4 + Storybook + design.md matter here
shadcn/ui v4 gives you accessible, customizable components out‑of‑the‑box via its CLI, preserving your ability to adapt design tokens later.

Storybook not only develops components in isolation but acts as your visual testing environment, ensuring the design system rules (loading, errors, a11y) are visibly enforced before they reach the app.

design.md is the single living document that both developers and AI refer to. By serving it through MCP, you guarantee the AI never suggests a component that violates the UI principles (e.g., raw alerts instead of the design‑system Toast).

The combination ensures that from day one, every screen is built from pre‑tested, documented, and contract‑aligned building blocks—catching visual regressions, accessibility flaws, and design violations automatically.
